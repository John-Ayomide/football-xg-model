Every code cell explained

Cell 1 — Load competitions
from statsbombpy import sb competitions = sb.competitions() print("Total competitions:", len(competitions))
What this does: sb.competitions() calls the StatsBomb API and returns a DataFrame of every competition and season in their open dataset.

The NoAuthWarning: You will see a warning about credentials. This is completely normal — it simply confirms you are using the free open data tier rather than a paid credentials account. Ignore it entirely.

Why this matters: Before loading any data you need the competition_id and season_id — these are the unique identifiers for every dataset call. Without them you cannot load matches or events.
Cell 2 — Find La Liga IDs
la_liga = competitions[competitions['competition_name'] == 'La Liga'] print(la_liga[['competition_id', 'season_id', 'season_name']])
What this does: Filters the competitions DataFrame to only La Liga rows, then shows the competition_id and season_id for each available season.

What we found: competition_id = 11 for La Liga. Season 2020/2021 has season_id = 90 — 18 seasons available going back to 1973/1974.

Why 2020/2021? It is the most recent season in the open dataset with the richest, most complete event data. It also includes Messi's final Barcelona season.
Cell 3 — Load matches
matches = sb.matches(competition_id=11, season_id=90) print("Total matches:", len(matches))
What this does: Loads all match metadata for La Liga 2020/2021 — one row per match with home/away teams, scores, dates, stadiums, referees, and managers.

Result: 35 matches. Note this is not a full 380-match season — StatsBomb's open data covers a subset of matches. 35 matches gives us 839 shots which is sufficient to train a reliable xG model.

Why match metadata matters: We use the match_id column to load detailed event data for each individual match in the next step.
Cell 4 — Extract all shot events
all_shots = [] for i, match_id in enumerate(matches['match_id']): events = sb.events(match_id=match_id) shots = events[events['type'] == 'Shot'].copy() shots['match_id'] = match_id all_shots.append(shots) shots_df = pd.concat(all_shots, ignore_index=True)
What this does: Loops through all 35 match IDs, loads every event for each match, filters to shots only, and combines into one DataFrame.

What is an event? StatsBomb records every single action as an event — passes, dribbles, tackles, shots, goalkeeper actions. A typical match has 3,000+ events. We filter to shots only because our xG model only needs shot data.

The loop pattern: We append each match's shots to a list then use pd.concat() at the end — this is significantly faster than concatenating inside the loop because it only touches memory once.

111 columns per shot: Each shot contains location coordinates, technique, body part, assist type, defensive context, and StatsBomb's own xG value — everything we need to build our model.

"Notebook 2: Feature engineering — shot distance, angle, 10 xG features created"

Every code cell explained

Parse x/y coordinates
shots_df['x'] = shots_df['location'].apply(lambda loc: loc[0]) shots_df['y'] = shots_df['location'].apply(lambda loc: loc[1])
What this does: The StatsBomb location column stores coordinates as a Python list like [85.3, 34.2]. We need to extract the x and y values separately as individual numeric columns.

lambda loc: loc[0] — a small anonymous function that takes a location list and returns the first element (x coordinate). loc[1] returns the second element (y coordinate).

StatsBomb pitch dimensions: The pitch is 120 x 80 yards. x=0 is the left goal line, x=120 is the right goal line. y=0 is the bottom touchline, y=80 is the top touchline. The goal centre is at approximately x=120, y=40.
Calculate shot distance
GOAL_X, GOAL_Y = 120, 40 shots_df['distance'] = np.sqrt( (shots_df['x'] - GOAL_X)**2 + (shots_df['y'] - GOAL_Y)**2 )
What this does: Calculates the straight-line distance from each shot location to the centre of the goal using the Pythagorean theorem: distance = √((x2-x1)² + (y2-y1)²).

Why distance matters for xG: Distance is one of the strongest predictors of goal probability. A shot from 5 yards scores far more often than one from 30 yards — even accounting for technique. Research shows goal probability roughly halves every 10 yards further from goal.

Why goal centre at (120, 40)? StatsBomb's pitch is 120x80 yards. The goal is on the right side at x=120. The goal width spans y=36 to y=44, so the centre is y=40.
Calculate shot angle
shots_df['angle'] = np.arctan2( shots_df['y'] - GOAL_Y, shots_df['x'] - GOAL_X ) * (180 / np.pi) shots_df['angle'] = shots_df['angle'].abs()
What this does: Calculates the angle between the shot location and the goal centre using arctan2 — converting from radians to degrees by multiplying by 180/π.

Why angle matters for xG: A central shot directly in front of goal has a small angle (close to 0°) giving the shooter the full width of the goal. A wide shot from a tight angle has a large angle, making the goal appear much narrower. Central shots score significantly more often than wide angle shots.

abs(): We take the absolute value because angles to the left and right of goal are equally difficult — we only care about the magnitude, not the direction.
Create target variable and binary features
shots_df['is_goal'] = (shots_df['shot_outcome'] == 'Goal').astype(int) shots_df['is_header'] = (shots_df['shot_body_part'] == 'Head').astype(int) shots_df['under_pressure'] = shots_df['under_pressure'].fillna(False).astype(int)
is_goal: Our target variable. 1 = shot was a goal, 0 = shot was not a goal. This is what the model learns to predict.

is_header: Headers score at a lower rate than foot shots on average — approximately 8% vs 15%. Including this feature helps the model account for that difference.

under_pressure: Shots taken under immediate defensive pressure score less often. StatsBomb records this as True/False — we convert to 1/0 for the model. fillna(False) handles cases where no pressure was recorded.
Purpose of Day 3
Feature engineering is where football knowledge meets mathematics. The raw StatsBomb data gives us coordinates — but the model needs meaningful features derived from those coordinates. Calculating distance and angle transforms raw x/y positions into features that directly represent shot quality. This is the most analytically interesting day of the project.
Football analytics context
The two most important features in any xG model are distance and angle. Every commercial xG model — StatsBomb, Opta, Wyscout — uses these as their foundation. Some advanced models add tracking data like goalkeeper position and defensive pressure. We start with the fundamentals and build up, which is exactly how professional analysts develop models.
Alternative approaches
More advanced xG models use postshot xG — which also considers where the goalkeeper was and the shot placement within the goal. Tracking data xG models use the position of every player on the pitch at the moment of the shot. We build a pre-shot model first — the industry standard starting point — before exploring these advanced approaches.

This is the core of the entire project. Training the actual xG model and comparing it against StatsBomb's own xG values is what proves the model works. Any candidate can say they "understand xG." Building one, validating it against an industry baseline, and explaining the methodology gap is what gets you hired.
Why Logistic Regression first?
Logistic regression is the original xG modelling method — it directly outputs a probability between 0 and 1 which is mathematically interpretable. The coefficient on distance tells you exactly how much goal probability drops per yard. XGBoost captures non-linear relationships but is harder to interpret. Using both and comparing them shows methodological rigour.
